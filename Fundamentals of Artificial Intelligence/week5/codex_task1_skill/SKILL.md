# 电商商品质检助手（Task 1）Codex Skill

## 目的
把 `week5/project_one` 包装成一个可演示的小产品：
1. 基于 CLIP 做**零样本图片分类**，完成“完好 / 外包装破损 / 空盒”三类识别。
2. 按课程要求输出 **Top-3 标签 + 概率 + 系统建议**。
3. 把系统拆成**后端服务**与**前端页面**：
   - 后端运行在本机，负责加载模型和执行推理。
   - 前端可以在另一台电脑访问，上传图片并拿到识别结果。
4. 补齐 README、依赖、测试脚本、评测脚本，便于课堂展示。

## 适用场景
当用户在仓库中提到以下任务时，主动使用本 skill：
- CLIP 图片打标签
- 第 5 周实践项目
- 电商商品质检助手
- week5 / project_one / task1
- 前后端分离部署 / 局域网访问 / 上传图片识别

## 先理解任务，不要误判
该项目核心是 **CLIP 零样本分类**，不是从零训练分类器。先检查代码是否已经使用：
- `transformers.pipeline(task="zero-shot-image-classification")`
- `openai/clip-vit-base-patch32`

若已经是上述形式，则本项目的重点不是重训练，而是：
1. 设计更好的标签描述词。
2. 提升界面与接口。
3. 做测试集评估。
4. 完成前后端分离部署。

只有当仓库中明确存在额外“训练脚本 / 数据集标注 / 微调需求”时，才增加训练流程。

## 任务目标（按优先级）
### P0 必做
- 找到项目入口代码。
- 将标签改为任务 1 所需三类：
  - 完好商品
  - 外包装破损商品
  - 缺少零件的空盒商品
- 输出 Top-3 概率。
- 根据 Top-1 给出建议：
  - 完好 -> 建议重新上架
  - 破损 / 空盒 -> 建议直接退款或人工复核
- 保证后端能被另一台电脑访问。

### P1 应做
- 把推理逻辑抽成独立函数。
- 提供 `FastAPI` 后端接口，例如 `/health`、`/predict`。
- 提供一个简单前端页面，可上传图片并显示结果。
- 提供最小化 README，写清启动方式。

### P2 加分
- 增加批量测试脚本。
- 统计 Top-1 / Top-3 准确率。
- 支持中英双语标签对比。
- 记录失败案例输出为 markdown 或 csv。

## 操作步骤
### 第一步：扫描项目
在仓库根目录执行：
- 查看目录结构
- 找到 `week5/project_one`
- 找到现有 python 文件、requirements、前端文件、说明文档

优先检查：
- 是否已有 `gradio` 页面
- 是否已有模型加载代码
- 是否已有测试图片目录
- 是否已有标签词表

### 第二步：改造成“可维护结构”
如果当前只有单文件脚本，优先整理为：
- `backend/app.py`：FastAPI 启动入口
- `backend/model_service.py`：模型加载与推理
- `frontend/index.html`：上传页面
- `scripts/eval.py`：测试集评估
- `requirements.txt`
- `README.md`

若仓库已有较成熟结构，则尽量少改，按现有风格补充。

### 第三步：实现后端
后端应满足：
- 启动时只加载一次 CLIP 模型
- 提供 `POST /predict`
- 输入：图片文件
- 输出 JSON：
  - `top3`
  - `top1_label`
  - `top1_score`
  - `advice`
- 提供 `GET /health`
- 允许跨域访问（CORS），便于另一台电脑前端调用
- 默认监听 `0.0.0.0`

### 第四步：实现前端
前端应满足：
- 可在浏览器中上传图片
- 调用后端 API
- 展示 Top-3 结果
- 展示系统建议
- 提示用户如何填写后端 IP

### 第五步：标签设计
优先使用**具体、视觉上可区分**的描述词，不要只写抽象词。
建议先从这些候选中试验：
- `a photo of an intact returned product with complete packaging`
- `a photo of a returned product with damaged outer packaging`
- `a photo of an empty box missing the product or accessories`

同时保留中文版本做对照实验：
- `一张完好无损、包装完整的退货商品照片`
- `一张外包装破损、挤压变形的退货商品照片`
- `一张商品缺失或配件缺失、只剩空盒的照片`

若用户未指定语言，优先在代码中同时保留中英版本，并在 README 中说明如何切换。

### 第六步：实现业务建议逻辑
规则建议：
- Top-1 为“完好”且分数 >= 0.55 -> `✅ 商品完好，建议重新上架`
- Top-1 为“外包装破损” -> `❌ 包装异常，建议人工复核或直接退款`
- Top-1 为“空盒/缺件” -> `❌ 疑似空盒或缺件，建议直接退款`
- 若最高分过低，例如 < 0.40 -> `⚠️ 结果不稳定，建议人工复核`

### 第七步：评测
如果项目里有测试集目录，优先读取；如果没有，则创建一个轻量评测约定：
- `data/test/intact`
- `data/test/damaged`
- `data/test/empty_box`

评测输出：
- 每张图的 Top-3
- Top-1 准确率
- Top-3 准确率
- 失败案例清单

## 两台电脑联调默认方案
- A 电脑：运行后端服务
- B 电脑：打开前端页面
- 前端中将 API 地址配置为 `http://A电脑IP:8000`
- A 电脑防火墙放行后端端口
- A/B 两台电脑处于同一局域网

## 开发原则
- 先让系统跑通，再优化结构。
- 尽量保留原有代码思路，不做无关重构。
- 不要凭空声称“训练完成”，除非真的新增了训练或微调流程。
- 若无法确认数据路径，写成可配置参数。
- 所有新增文件都要配清晰注释。
