# 路线图

## 阶段 1：更稳的展示原型

目标：让当前本地演示覆盖更广、更稳定、更容易讲清楚。

需要做：

- 将知识卡从 39 张扩展到 80-120 张。
- 替换早期占位来源 URL，改为已审核的官方或高质量来源。
- 给 `scripts/powershell/TEST_COMMANDS.ps1` 增加更多回归问题。
- 每次更新知识卡后保持 `scripts/validate_kb.py` 通过。
- 前端展示命中的 `card_id` 和分类，方便演示解释。
- 添加一个简单的覆盖面页面或接口，列出所有有效卡片。

退出标准：

- 至少 80 张有效卡片。
- 每张卡片都有来源引用和证据说明。
- 每张卡片的代表问题都能检索到目标卡片。
- 常见药物剂量和诊断类问题会被拦截。

## 阶段 2：结构化知识库

目标：从手工维护 JSON 逐步走向可维护的数据结构。

推荐路径：

- 卡片结构稳定前继续使用 JSON。
- 本地演示和课程展示阶段可迁移到 SQLite。
- 只有在需要多人编辑或云端部署时，再考虑 PostgreSQL。

建议表结构：

- `cards`：卡片内容、分类、子分类、标题、回答字段、状态。
- `question_patterns`：每张卡片的多个自然问法。
- `sources`：来源标题、URL、来源类型、审核状态。
- `card_sources`：卡片和来源的关联。
- `red_flags`：风险规则和升级提醒。
- `review_logs`：谁在什么时候审核了什么。

为什么不急着上数据库：

- 当前难点是卡片质量和人工审核，不是存储。
- JSON 更适合卡片结构仍在变化时快速迭代。
- 数据库太早引入会让每次内容编辑更慢。

## 阶段 3：审核和管理流程

目标：让知识更新更安全。

需要做：

- 细化审核状态：`draft`、`reviewed`、`needs_source`、`retired`。
- 添加发布前检查脚本，拒绝带占位 URL 的生产卡片。
- 添加简单后台表单或表格导入流程。
- 每张卡片保留审核备注。
- 将高风险主题和普通育儿建议分开维护。

退出标准：

- 新卡没有来源不能进入展示集。
- 每张风险升级卡都有明确红旗信号。
- 卡片修改历史可追溯。

## 阶段 4：更好的检索

目标：让自然语言问题更稳定地匹配正确卡片。

可选升级：

- 为卡片增加别名和同义词字段。
- 引入中文分词打分。
- 当卡片规模足够大后，再考虑向量检索。
- 可以添加重排序，但分类过滤和安全过滤必须先执行。

重要规则：

- 检索不能为了避免“未命中”而返回无关卡片。

## 阶段 5：微信小程序

目标：把当前 FastAPI 服务作为微信小程序后端。

需要做：

- 保持 `/api/ask` 作为主要后端接口。
- 部署 HTTPS，因为小程序生产请求需要配置安全域名。
- 在微信公众平台配置小程序 request 合法域名。
- 构建小程序页面：问题输入、回答展示、来源展示、反馈。
- 在小程序界面固定展示免责声明和隐私说明。
- 增加轻量日志和反馈收集。

开发捷径：

- 本地开发时，微信开发者工具可以设置跳过域名校验。
- 真实用户测试时，必须使用 HTTPS 和已配置的合法域名。

## 阶段 6：部署和运维

目标：让开发电脑之外的人可以稳定访问。

演示选项：

- 同一 Wi-Fi 演示：Uvicorn 绑定 `0.0.0.0`，分享局域网 IP。
- 临时公网演示：使用 ngrok 等隧道工具。
- 更稳定的公网演示：部署到云服务器并配置 HTTPS。

生产需要：

- HTTPS。
- 认证或防滥用控制。
- 错误日志。
- 带隐私保护的请求日志。
- 知识数据备份和版本管理。
- 服务可用性监控。

## 阶段 7：接近真实使用的完整形态

这不只是“多写代码”。

还需要：

- 已审核的知识来源。
- 清晰的安全边界。
- 内容维护流程。
- 用户反馈闭环。
- 稳定部署。
- 隐私与合规检查。
- 保守且可审计的医疗风险升级机制。
